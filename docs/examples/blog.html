Building Scalable Design Systems with Semantic CSS

Why naming things well matters more than you think, and how a semantic-first approach can transform your component library.

When we started building our component library, we faced a choice that every frontend team encounters: utility-first or semantic classes? After months of iteration, we landed on a hybrid approach that gave us the best of both worlds.

The Problem with Utility-Only

Utility classes are powerful for prototyping, but they create maintenance challenges at scale. Consider a button that appears in 47 places across your app:

<!-- Before: utility soup -->
<button class="px-4 py-2 bg-blue-500 text-white
  rounded-lg hover:bg-blue-600 active:scale-95
  transition-all duration-200 font-medium
  text-sm shadow-sm">
  Save Changes
</button>

<!-- After: semantic -->
<button class="btn">Save Changes</button>

The semantic version is not just shorter — it's a contract. When you change how buttons look, you change it in one place. Every .btn in your codebase updates automatically.

Naming Conventions That Scale

Good names should be obvious to a newcomer. We follow three principles:

1. Use the HTML element name when possible (table, dialog).

2. Use established patterns from UI libraries (btn, card, badge).

3. Use modifiers as suffixes (btn-primary, badge-success).

Dark Mode for Free

When your components use CSS custom properties for all colors, dark mode becomes trivial. No class changes, no JavaScript — just swap the variable values:

:root {
  --background: oklch(1 0 0);
  --foreground: oklch(0.145 0 0);
}
.dark {
  --background: oklch(0.145 0 0);
  --foreground: oklch(0.95 0 0);
}

Every component that references var(--background) adapts instantly. Zero specificity wars, zero forgotten overrides.

The Result

After migrating to semantic classes, our team saw measurable improvements:

62%

Less CSS shipped

3x

Faster onboarding

89%

Dev satisfaction

Semantic CSS isn't a step backward — it's the natural evolution of design systems. Name things well, use CSS variables for theming, and let the browser do the heavy lifting.
